# Power Apps Code Apps — Architecture Checklist

Ground truth: the CodeSpec starter (`voyager163/codespec`, main branch, `templates/starter/`) and the Power Apps Code Apps model. Check every rule; report each as PASS / FAIL / N-A with file:line evidence.

## A. Project foundation (Blockers if missing)

| # | Rule | How to check |
|---|---|---|
| A1 | `power.config.json` exists at repo root, generated by `pac code init` — never hand-written | File present; if absent, remediation is `pac code init` via codeapps-app-scaffolder |
| A2 | `@microsoft/power-apps` in `dependencies` | `package.json` |
| A3 | `@microsoft/power-apps-vite` in `devDependencies` and `powerApps()` plugin registered in `vite.config.ts` | `vite.config.ts` plugins array |
| A4 | Host initialization: a `PowerProvider` (or equivalent) calls `initialize()` from `@microsoft/power-apps/app` before the app renders | Grep for `initialize(` |
| A5 | Vite + React + TypeScript with `tsc -b` in the build script; TS `strict: true` | `package.json` scripts, `tsconfig` |
| A6 | Buildable and locally testable: `npm run build` succeeds, `npm run dev` serves | Run them |

## B. Data access (Blockers)

| # | Rule | How to check |
|---|---|---|
| B1 | All runtime data access goes through generated services under `src/generated/` (from `pac code add-data-source`) or Power Platform connectors | Grep imports of `src/generated`; compare against every network call found in C |
| B2 | No custom backend server (Express, Fastify, serverless functions called at runtime) | Repo tree, `package.json` |
| B3 | Dataverse is the system of record; local mock data is dev-only and excluded from production paths | Grep `mock` usage in components |

## C. Forbidden — external runtime services (Blockers)

npm packages that bundle into `dist/` are **allowed**. What is forbidden is the app *talking to services outside Power Platform at runtime*:

| # | Rule | How to check |
|---|---|---|
| C1 | No `fetch`/`axios`/`XMLHttpRequest`/`WebSocket`/`EventSource` calls to non-Power-Platform origins | Grep each token; resolve the target origin |
| C2 | No CDN `<script>` or `<link>` tags in `index.html` (scripts, styles, fonts) — everything bundles into `dist/`; self-host fonts | Read `index.html` |
| C3 | No custom authentication: no login endpoints, no JWT/tokens in `localStorage`/`sessionStorage`, no password fields. The Power Apps host authenticates users | Grep `localStorage`, `token`, `login`, `password` |
| C4 | No third-party backend SDKs: firebase, supabase, aws-amplify, parse, appwrite, pocketbase, or third-party analytics beacons | `package.json` + grep imports |
| C5 | No secrets, API keys, or connection strings in source — use environment variables (codeapps-env-vars-specialist) | Grep `apiKey`, `secret`, `key:`, long literals |

## D. Conventions (Major)

| # | Rule |
|---|---|
| D1 | Source layout follows the starter: `src/components/` (with `ui/` primitives), `src/hooks/`, `src/lib/`, `src/pages/`, `src/generated/` |
| D2 | Path alias `@` → `./src` configured in both `vite.config.ts` and `tsconfig` |
| D3 | Routing stays client-side and host-safe (starter uses `react-router-dom`; avoid URL strategies that fight the host iframe) |
| D4 | Lint/format present and passing: ESLint (`--max-warnings 0`) and Prettier per starter config |
| D5 | Spec-driven development wired: `AGENTS.md` present, OpenSpec (`openspec/`) used for meaningful changes |

## E. Testing & verification (Major)

| # | Rule |
|---|---|
| E1 | `npm run dev` serves the app for localhost testing — this must survive every remediation |
| E2 | Unit tests run with Vitest (starter thresholds: 80% branches/functions/lines/statements where configured) |
| E3 | `pac code run` used for host-emulated testing before `pac code push` |

## Allowed — never flag these

- Any npm package that runs client-side and is bundled by Vite (React, Radix/shadcn, Tailwind, TanStack, recharts, zustand, date-fns, chart libraries, icon sets…)
- Dev-time tooling and localhost servers (Vite dev server, Vitest, Playwright)
- Calls to Power Platform / Dataverse endpoints via generated services and connectors
